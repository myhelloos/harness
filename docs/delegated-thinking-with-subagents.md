# 委派式思考：子智能体与 Skill 协作指南

检查日期：2026-07-12 Asia/Shanghai

本文整理本次 session 中关于子智能体、Skill、架构原则和现代 AI 工程设计模式的讨论，目标是形成一份可直接用于本 workspace 的委派式思考指南。

这里的“委派式思考”不是把判断责任交给更多 agent，而是把复杂问题拆成边界清晰、证据可追、结果可合并、失败可记录的认知任务。

## 核心模型

委派式思考的基本分工：

```text
主智能体：判断问题结构、分配注意力、控制风险、合并结果。
子智能体：在清晰边界内完成局部认知、探索、审查或执行任务。
Skill：沉淀可复用的方法、流程、约束、工具路由和产物格式。
```

子智能体的价值不在于“多几个模型一起想”，而在于管理复杂度：

- 隔离上下文，避免一个推理链路承载所有细节。
- 并行探索，降低单一路径误判带来的盲区。
- 专门化任务，让不同子任务按不同证据标准执行。
- 分离生成与审查，避免同一 agent 无约束地自我确认。
- 保留失败路径，把失败也变成可分析证据。

## 两种协作方向

### 子智能体预加载 Skill

这是“先选人，再给工具包”。

```text
主智能体
  -> 创建子智能体 A
  -> 给 A 预加载 Skill: research / code-review / benchmark
  -> A 独立完成一个子任务
  -> 主智能体验收和合并
```

重点是：子智能体是主体，Skill 是能力配置。

适合场景：

- 任务是探索型、临时型、开放型。
- 还不知道稳定流程是什么。
- 需要多角度比较、并行调研或局部审查。
- 子任务有清楚边界，但执行路径需要判断。
- Skill 只是让子智能体更像某类专家，而不是规定完整流程。

例子：

- 让一个预加载 research Skill 的子智能体核验官方文档。
- 让一个预加载 code-review Skill 的子智能体审查某组 diff。
- 让一个预加载 benchmark Skill 的子智能体设计小型评测探针。

风险：

- 如果上下文和验收标准不清楚，子智能体会产生长篇但不可合并的输出。
- 如果权限过宽，探索型委派可能越过任务边界。
- 如果缺少验证，多个子智能体可能只是制造更多未经审查的观点。

### Skill 派生子智能体

这是“先进入流程，再按流程派人”。

```text
主智能体
  -> 选择 Skill: benchmark-evaluator
      -> Skill 派生 source-checker 子智能体
      -> Skill 派生 runner 子智能体
      -> Skill 派生 analyst 子智能体
      -> Skill 汇总证据并产出报告
```

重点是：Skill 是流程主体，子智能体是执行部件。

适合场景：

- 流程已经反复出现。
- 每次都需要相同证据、产物、目录结构或验证步骤。
- 失败路径也需要被记录和分类。
- 子智能体之间的分工是流程设计的一部分。
- 希望把经验固化为可复用的 SOP / Playbook / Runbook。

例子：

```text
benchmark Skill:
  -> source-checker：核验版本、日期、官方来源。
  -> runner：执行命令，保留原始日志。
  -> analyst：区分观察、推断和建议。
  -> reporter：生成最终实验报告。
```

风险：

- 过早固化会让流程僵硬，压制探索。
- Skill 过重会破坏渐进式披露，把过多背景塞进上下文。
- 子智能体数量增加后，如果没有统一证据格式，汇总成本会上升。

## 选择规则

一句话判断：

```text
如果主要是在“给一个子任务找合适的人”，用子智能体预加载 Skill。
如果主要是在“把一个稳定流程产品化”，用 Skill 派生子智能体。
```

| 判断维度 | 子智能体预加载 Skill | Skill 派生子智能体 |
| --- | --- | --- |
| 控制权 | 主智能体调度 | Skill 流程调度 |
| 任务形态 | 探索性、开放性、临时性 | 稳定、重复、可模板化 |
| Skill 角色 | 能力包 / 专家设定 | 工作流定义 |
| 子智能体角色 | 独立专家 | 流程中的执行单元 |
| 适合阶段 | 早期探索 | 成熟沉淀 |
| 输出一致性 | 中等，依赖委派契约 | 高，依赖 Skill 规范 |
| 灵活性 | 高 | 中等 |
| 复用方式 | 复用专家配置 | 复用完整流程 |

推荐演进路径：

```text
手动思考
  -> 临时委派给子智能体
  -> 给子智能体预加载 Skill
  -> 总结重复出现的流程
  -> 固化为 Skill
  -> Skill 内部派生子智能体
  -> 形成可复现 harness
```

## 委派前检查

每次委派前先回答六个问题：

1. 这个任务能否被明确交付？
2. 子智能体需要哪些上下文？
3. 它能否独立验证结果？
4. 失败是否可接受，是否需要先限制为分析或草案？
5. 结果如何合并回主线？
6. 这个委派模式是否已经重复到值得沉淀为 Skill？

推荐上下文最小集：

```text
目标：要完成什么。
边界：需要做什么，不需要做什么。
输入：相关文件、资料、日志、已有结论。
约束：时间、预算、权限、禁止事项。
输出：结构、格式、粒度、必须包含的证据。
验收：如何判断完成，如何处理无法验证。
```

## 委派模板

可以把下面模板作为主智能体派发子任务的默认格式：

```text
角色：
你是负责 [具体职责] 的子智能体。

目标：
完成 [具体目标]。

上下文：
- 项目背景：
- 当前约束：
- 相关文件/资料：
- 已知事实：

任务边界：
- 你需要做：
- 你不需要做：
- 不要修改/不要假设：

输出格式：
- 结论：
- 依据：
- 风险：
- 未确认事项：
- 建议下一步：

验证要求：
- 必须检查：
- 如果无法验证，明确说明原因。
```

对本 workspace，额外建议加入：

```text
证据要求：
- 外部事实需要来源 URL 和检查日期。
- 实验需要记录命令、环境、原始日志和已知限制。
- 评测需要分离任务定义、评分方法、原始日志和分析。
- 宽泛结论必须说明适用的工作负载、预算和风险画像。
```

## 常见委派类型

### Research 委派

用于官方文档核验、论文查证、工具能力确认和现状调研。

输出要求：

- 来源链接。
- 检查日期。
- 观察结果。
- 与当前任务的关系。
- 不确定性和过期风险。

### Review 委派

用于代码审查、设计审查、实验审查和文档审查。

输出顺序：

```text
严重问题
中等风险
小问题
测试或证据缺口
开放问题
```

Review 委派应优先发现问题，而不是先总结优点。

### Runner 委派

用于执行实验、测试、benchmark 或可复现探针。

输出要求：

- 命令。
- 环境。
- 原始结果。
- 失败日志。
- 初步观察。
- 已知限制。

Runner 不应把一次运行直接包装成稳定结论。

### Analyst 委派

用于解释结果、比较方案、提炼模式。

输出必须区分：

```text
观察：直接来自日志、代码或来源的事实。
推断：基于观察得出的解释。
建议：下一步行动。
证据强度：强、中、弱或需要复验。
```

### Synthesizer 委派

用于整合多个子智能体输出。

Synthesizer 的职责不是重新发明事实，而是：

- 检查冲突。
- 压缩冗余。
- 标注证据强弱。
- 把分散结果变成可决策材料。
- 明确哪些结论还不能成立。

## 主智能体不可外包的责任

主智能体可以委派局部任务，但不能外包最终责任：

- 最终决策。
- 风险接受。
- 权限授予。
- 跨任务取舍。
- 结果合并。
- 冲突裁决。
- 对用户的最终承诺。

子智能体可以提供候选答案、证据和风险说明，但主智能体必须负责最终判断和交付质量。

## 现代 AI 工程中的重要设计模式

委派式思考和现代 AI 工程中的架构原则是同一类问题。优秀架构原则具有跨领域普适性，因为复杂系统会反复遇到相似矛盾：耦合、边界、状态、反馈、失败、演化和权限。

### Orchestrator / Worker

主控负责分解、调度、预算、失败处理和合并；worker 负责局部执行。

```text
Orchestrator
  -> Research Worker
  -> Runner Worker
  -> Review Worker
  -> Synthesizer Worker
```

### Tool-First Agent

模型负责判断，工具负责执行。

适合交给工具的能力：

- 搜索和资料获取。
- 文件修改。
- 测试和命令执行。
- 结构化解析。
- 数据查询。
- 截图、日志和 artifact 收集。

### Context Engineering

上下文窗口是稀缺资源，不是数据库。

委派时应明确：

- 哪些信息必须进入子智能体上下文。
- 哪些信息只给路径或索引。
- 哪些历史需要压缩。
- 哪些证据必须原样保留。
- 哪些状态应该结构化保存。

### Evaluator / Critic

生成和评估应分离。

```text
Generator -> Candidate Output
Evaluator -> Correctness / Risk / Coverage Check
Refiner -> Fix Based on Findings
```

在 harness 实验中，evaluator 可以是测试、lint、静态分析、评分器、第二个 agent 或人工审查。

### Structured Intermediate Representation

复杂任务不应只在自然语言里流动。中间状态越结构化，越容易验证、重放、审计和跨 agent 传递。

示例：

```json
{
  "task": "compare_agent_harnesses",
  "requires_sources": true,
  "outputs": ["source_map_update", "experiment_log", "analysis_note"],
  "validation": ["official_docs_checked", "commands_recorded"]
}
```

### Capability-Based Security

不要只问“这个 agent 是否可信”，而要问：

```text
它被授予了哪些能力？
这些能力能访问什么？
能写哪里？
能不能联网？
能不能执行命令？
能不能删除文件？
高风险动作是否需要审批？
```

权限边界必须由系统保证，不能只靠 prompt 要求模型自觉。

### Progressive Autonomy

自主性应随验证能力和风险画像逐步提升：

```text
建议
  -> 草案
  -> 可回滚修改
  -> 低风险自动执行
  -> 高风险前请求审批
```

不要因为模型看起来聪明就直接授予高风险能力。

## 在本 workspace 的落地方式

### 何时创建实验

当一个委派模式需要验证其有效性时，在 `experiments/` 下创建子目录，并包含：

- `README.md`
- 安装或准备命令
- 运行命令
- 结果笔记
- 已知限制

适合实验的问题：

- 预加载 Skill 的 reviewer 是否比普通 reviewer 更稳定？
- Research / Runner / Analyst 三段式委派是否提升失败归因质量？
- Skill 派生子智能体是否能降低主智能体上下文负担？
- 多 agent 并行探索是否真的提高通过率，还是只是增加成本？

### 何时创建评测

当要比较不同 harness、不同模型、不同委派拓扑时，放入 `evals/`。

评测必须分离：

- 任务定义。
- 评分方法。
- 原始日志。
- 分析。

最小评测轴沿用本仓库 `docs/radar.md`：

- 成功。
- 成本。
- 可靠性。
- 可调试性。
- 控制。
- 可移植性。
- 可维护性。

### 何时沉淀为 Skill

当同类委派重复出现，并且每次都需要相似步骤、证据和产物时，考虑沉淀为 Skill。

判断标准：

- 触发条件是否清楚？
- 负触发条件是否清楚？
- 产物是否稳定？
- 失败路径是否可描述？
- 是否有 references / scripts / assets 可以渐进式披露？
- 是否需要关闭隐式触发或要求人工审批？

## 反模式

- 把“多 agent”当作智能本身，而不是复杂度管理手段。
- 委派“想想这个问题”，却不给交付物和验收标准。
- 给子智能体过多上下文，让它继承主线程所有噪音。
- 给子智能体过少上下文，让它靠猜测补齐关键事实。
- 让同一个 agent 既生成方案又作为唯一评审。
- 把模型自评当作唯一评分器。
- 在单 agent 循环尚不可度量前，就引入复杂多 agent 编排。
- 将探索性流程过早固化为重量级 Skill。
- 用 prompt 约束替代 sandbox、approval、tool whitelist 和日志审计。

## 最小实践清单

每次委派至少留下这些信息：

- 为什么委派。
- 委派给哪个角色。
- 给了哪些上下文。
- 允许做什么，不允许做什么。
- 预期输出格式。
- 如何验证。
- 结果如何合并。
- 有哪些失败、冲突或不确定性。

如果这些信息无法写清楚，通常说明任务还不适合委派；应先由主智能体继续澄清问题结构。

## 一句话原则

```text
好的委派式思考，是把复杂问题拆成边界清晰、证据可追、结果可合并、失败可记录的认知任务。
```

在本 workspace 中，子智能体、Skill、工具、评测和实验都应服务于这个目标：让 AI agent harness 的能力变得可复现、可审查、可演进。
